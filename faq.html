◇ Straight answers, including the uncomfortable ones

Questions a skeptic asks.

This is wallet-custody software. The honest failure modes are here, not buried.

Who controls the private key?

auth · 01

Command-parsing and Telegram-handling code never touch the key directly. It lives behind an isolated Signer module. Today that's a local AES-256-GCM encrypted keystore (devnet/testnet); the same interface is built to swap in a real KMS/HSM custody backend, and the config loader hard-refuses to start against mainnet until that swap happens.

What if my Telegram is compromised?

risk · 02

Straight answer: today access is the numeric-ID allowlist alone — there's no second factor yet. But no command in this build signs a transfer. The only thing that moves funds is the automatic sweep + cold-forward, and those go to preconfigured destinations, not chat-chosen ones — a compromised allowlisted account can't redirect them by typing. If your Telegram is compromised, tell the operator to drop your ID from the allowlist. A burn-on-use TOTP and an Argon2 login are the planned second factor for when the splitter lands.

Why Telegram at all?

platform · 03

It's the interface most admins already have installed, and works equally well in a DM or a group. It's not the security boundary, though — access is gated by a separate numeric Telegram-ID allowlist regardless of which chat app is in front of it. A password and OTP layer are planned on top of that gate.

Is this on mainnet?

net · 04

No — devnet only, enforced in code, not just policy: the config loader refuses to start against mainnet-beta unless a real KMS/custody signer replaces the local keystore. An internal security review plus an independent test pass already ran at this stage; both found and fixed real issues before this page existed.

How tamper-proof is the log, really?

log · 05

Every command attempt (allowed or denied) is written to a local append-only log where each entry's hash includes the previous entry's hash. Editing or deleting a past line breaks the chain for everything after it, and that break is detectable by re-verifying. It runs on the same machine as the bot, not a decentralized store — it makes tampering detectable, not physically impossible for someone with direct filesystem access.

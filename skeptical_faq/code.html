Who controls the private key?

AUTH_01

Command-parsing and Telegram-handling code never touch the key directly. It lives behind an isolated Signer module. Today that's a local AES-256-GCM encrypted keystore file (devnet/testnet); the same interface is built to swap in a real KMS/HSM custody backend, and the bot's config loader hard-refuses to start against mainnet until that swap happens.

What happens if Telegram is compromised?

RISK_02

Straight answer: a compromised, already-logged-in session with a live OTP code can still authorize a fund-moving command. This is wallet-custody software, not a Telegram-account-security product. What limits the damage: every fund-moving command needs a fresh OTP even mid-session, and that code is burned after one use, so a captured code can't be replayed a second time. If your Telegram account is compromised, treat your admin password and OTP secret as compromised too and rotate them immediately.

Why Telegram at all?

PLAT_03

It's the interface most admins already have installed, and works equally well in a DM or a group for status updates. It's not the security boundary, though. Access is gated by a separate Telegram-ID allowlist, password, and OTP layer regardless of which chat app is in front of it.

Is this on mainnet?

NET_04

No, devnet only, and that's enforced in code, not just policy: the config loader refuses to start against mainnet-beta unless a real KMS/custody signer is configured in place of the local keystore. An internal security review plus an independent test pass already happened at this stage. Both found and fixed real issues before this page existed.

Tamper-proof log details

LOG_05

Every command attempt (allowed or denied) is written to a local append-only log where each entry's hash includes the previous entry's hash. Editing or deleting a past line breaks the chain for everything after it, and that break is detectable by re-verifying the chain. This runs on the same machine as the bot, not a separate decentralized store: it makes tampering detectable, not physically impossible for someone with direct filesystem access.

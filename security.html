The security model.

Four layers are enforced in the current build. Below them, in the same type size, is what's still planned and what they deliberately do not cover.

01 / CONTROLS

What's enforced today.

Layer 01 · Identity

Numeric-ID allowlist

Commands run only for pre-registered Telegram user IDs — never @usernames alone, which can change hands. Set in config before the bot runs; there's no in-bot request-access path.

Layer 02 · Custody

Isolated signer boundary

Chat handling and parsing never touch key material. Signing sits behind one Signer interface — local AES-256-GCM keystore now, KMS/HSM for mainnet.

Layer 03 · Routing

Slippage + min-output gates

Every sweep is bounded by a slippage ceiling and a minimum-output floor. A route that would deliver less than the floor is skipped before it's signed — not unwound afterward.

Layer 04 · Evidence

Hash-chained audit log

Every attempt, allowed or denied, is appended with a hash folding in the prior entry. Tampering with any past line breaks verification for all lines after it.

Guard · Network

Mainnet-refuse guard

The config loader hard-refuses to start against mainnet-beta with a local keystore. Real funds can't run on a hot key by accident.

Guard · Hot wallet

Cold-forward drain

Swept SOL above a set buffer forwards to a cold wallet, so the Telegram-operated hot wallet never holds more than it needs to.

02 / UPCOMING

Planned controls — not yet live.

Planned

Argon2 password login

/login opening a time-boxed session, password Argon2-hashed and never stored in plaintext. Today access is the allowlist only.

Planned

Single-use TOTP

A fresh 6-digit code per fund-moving command, burned on first use with global replay tracking. Not in this build.

Planned

Group secret scrub

Auto-delete of any password or OTP typed into a group by mistake, and immediate burn of the code, rather than leaving it in the transcript.

03 / LIMITS

What it doesn't protect against.

ScenarioRealityWhat limits it
Allowlisted account takeoverToday access is the allowlist alone — no second factor. A compromised allowlisted account can issue commands, though no command signs a transfer in this build.Only automatic sweep + cold-forward move funds; those go to preconfigured destinations, not chat-chosen ones. TOTP + login are the planned second factor.
Host filesystem accessSomeone with direct disk access can read/rewrite files.Audit chain makes tampering detectable, not impossible. Keystore is encrypted at rest.
Telegram account securityThis is custody software, not account-security software.Keep your Telegram account secured; the operator can remove a compromised ID from the allowlist.
Mainnet with local keyNot allowed. Local keystore is devnet/testnet only.Config loader hard-refuses to start against mainnet-beta without a custody signer.

An internal security review and an independent test pass have both run at this stage. Both found and fixed real issues. Independent third-party audit before mainnet remains pending.
